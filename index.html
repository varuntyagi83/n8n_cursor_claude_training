Use the official czlonkowski/n8n-mcp package.

Restart Cursor & Verify

Restart Cursor for changes to take effect, then verify the connection.

Windows (PowerShell)

claude mcp add-json "n8n-mcp" '{ "command": "npx", "args": ["n8n-mcp"], "env": { "MCP_MODE": "stdio", "LOG_LEVEL": "error", "DISABLE_CONSOLE_OUTPUT": "true" } }'

To manage workflows directly, add your n8n API credentials.

Full Configuration with n8n API
claude mcp add-json "n8n-mcp" '{ "command": "npx", "args": ["n8n-mcp"], "env": { "MCP_MODE": "stdio", "LOG_LEVEL": "error", "DISABLE_CONSOLE_OUTPUT": "true", "N8N_API_URL": "https://your-n8n-instance.com", "N8N_API_KEY": "your-api-key-here" } }'

Verify the Connection

25 ‚Äî Best Practices

Tips for Success with AI Coding

Hard-won lessons from building 100+ projects

üéØ

Be Specific

"Add a button" ‚Üí Bad
"Add a blue CTA button with text 'Get Started', rounded corners, hover animation, that scrolls to #contact section" ‚Üí Good

üîç

Review Everything

AI makes mistakes. Always read the code before accepting. Look for hardcoded values, missing error handling, security issues.

üì¶

Small Changes

Don't ask for a whole app at once. Break it into small pieces: header, hero, features, etc. Easier to review, easier to fix.

üíæ

Commit Often

After each working change, commit. If AI breaks something, you can always go back. "git commit -m 'Add hero section'"

üìñ

Use Context

Reference existing files with @filename. Give AI the full picture. "Match the style in @hero.css when creating @features.css"

üß™

Test Immediately

After each change, check it works. Open in browser, click buttons, submit forms. Catch bugs early, not after 10 more changes.

26 ‚Äî Version Control

Branching & Small PR-Sized Changes

Professional workflow: never break main, always have a safety net

üå≥ What is Branching?
A branch is a parallel version of your code. Work on features without affecting the main (stable) version. When ready, merge back.
Basic Git Workflow
# Create a new branch git checkout -b feature/add-hero-section # Make changes with Cursor... # Stage and commit git add . git commit -m "Add hero section with CTA" # Push to remote git push origin feature/add-hero-section # Create PR on GitHub/GitLab # After review, merge to main

Branch Naming:

feature/ ‚Äî New features
fix/ ‚Äî Bug fixes
refactor/ ‚Äî Code cleanup
docs/ ‚Äî Documentation

Why Small PRs Matter:

‚úÖ Easier to Review

Reviewers (and you!) can actually read and understand 50 lines. 500 lines? Eyes glaze over.

‚úÖ Easier to Revert

Something broke? Revert one small change, not a massive commit with 20 features.

‚úÖ Faster Feedback

Get feedback on hero section before building 10 more sections wrong.

‚úÖ Less Merge Conflicts

Smaller changes = fewer conflicts when merging back to main.

üìè Rule of Thumb
If you can't describe your change in one sentence, it's probably too big. One feature = One branch = One PR.
